# Office Dispatch Workbench MVP

## 1. Purpose

This document defines the first internal MVP version of the Office Dispatch Workbench.

The goal is to turn the current dispatch optimizer project into an office-use order assignment system that can be used by office staff to manually enter delivery orders, generate driver and vehicle assignments, manually adjust the result, save the dispatch plan, lock the final plan, and review historical dispatch records.

This MVP is not intended to be a full ERP system, a WooCommerce integration, or a driver mobile app.

---

## 2. MVP Positioning

The first version is an internal office dispatch tool.

The system should support the following workflow:

```text
Create dispatch batch
→ Manually enter orders
→ Review available drivers and vehicles
→ Generate dispatch plan
→ Review automatic assignment
→ Manually adjust assignment if needed
→ Save draft
→ Lock final dispatch plan
→ View historical dispatch records
→ Export / print dispatch summary
```

The backend dispatch engine remains the authoritative planner.

The public output contract must remain:

```text
plans
order_assignments
exceptions
```

---

## 3. In Scope

The first MVP version includes:

1. Manual order entry
2. Office-only usage
3. Driver and vehicle management for dispatch planning
4. Automatic dispatch generation using the existing `DispatchEngine`
5. Driver-to-vehicle-to-orders assignment summary
6. Manual adjustment of generated assignments
7. Saving dispatch plans as drafts
8. Locking final dispatch plans
9. Read-only access to locked dispatch plans
10. Historical dispatch batch lookup
11. Basic validation and warning messages
12. Export or print-friendly dispatch summary

---

## 4. Out of Scope

The first MVP version does not include:

1. WooCommerce automatic order import
2. WooCommerce order status write-back
3. Driver mobile app
4. Customer notification
5. Automatic SMS or email dispatch notification
6. Live vehicle tracking
7. Detailed navigation route optimization
8. Full ERP functionality
9. Complex multi-role permission system
10. Accounting, invoicing, warehouse, or stock modules

These can be considered in later phases after the office dispatch workflow is stable.

## 5. Core Business Rules

### 5.1 Backend Authority

The backend `DispatchEngine` is the authoritative source for generated dispatch results.

The frontend may collect input, display results, and allow manual adjustments, but production dispatch generation should come from the backend API.

### 5.2 Output Contract

The top-level dispatch result must always contain:

```json
{
  "plans": [],
  "order_assignments": [],
  "exceptions": []
}
```

Do not replace this contract with another structure.

### 5.3 Manual Adjustment

Office staff must be allowed to manually adjust generated assignments.

Supported manual changes:

1. Change assigned driver
2. Change assigned vehicle
3. Assign an unassigned order
4. Remove an assignment
5. Add a manual adjustment reason
6. Save the adjusted result

Manual changes should be clearly marked as manual.

Recommended fields:

```text
assignment_source: AUTO | MANUAL
manual_reason: string
```

### 5.4 Locked Dispatch Plan

A locked dispatch plan represents the final approved dispatch result for a dispatch date.

Once locked:

1. Orders cannot be edited
2. Drivers and vehicles cannot be changed for that batch
3. Assignments cannot be modified
4. The plan cannot be regenerated and overwritten
5. The plan can still be viewed
6. The plan can still be exported or printed

If changes are required after locking, the recommended approach is to copy the locked plan into a new draft revision.

## 6. Dispatch Batch Status Flow

Each dispatch batch should have a status.

Recommended statuses:

```text
DRAFT
GENERATED
ADJUSTED
LOCKED
CANCELLED
```

### 6.1 DRAFT

Initial state.

Allowed actions:

1. Add orders
2. Edit orders
3. Edit available drivers
4. Edit available vehicles
5. Save draft
6. Generate plan

### 6.2 GENERATED

A plan has been generated by the backend dispatch engine.

Allowed actions:

1. Review generated result
2. View exceptions
3. Manually adjust assignments
4. Regenerate plan
5. Save
6. Lock

### 6.3 ADJUSTED

At least one manual adjustment has been made.

Allowed actions:

1. Continue manual adjustment
2. Save
3. Lock
4. Export / print
5. Regenerate only if the user confirms that manual adjustments may be overwritten

### 6.4 LOCKED

Final dispatch plan.

Allowed actions:

1. View
2. Export
3. Print
4. Copy to new draft

Not allowed:

1. Edit orders
2. Edit assignments
3. Regenerate plan
4. Delete result

### 6.5 CANCELLED

The dispatch batch is no longer active.

Allowed actions:

1. View only

## 8. Recommended Backend API

### 8.1 Dispatch Batch APIs

```text
GET /api/dispatch/batches
POST /api/dispatch/batches
GET /api/dispatch/batches/{batch_id}
PUT /api/dispatch/batches/{batch_id}
POST /api/dispatch/batches/{batch_id}/lock
POST /api/dispatch/batches/{batch_id}/copy
POST /api/dispatch/batches/{batch_id}/cancel
```

### 8.2 Order APIs

```text
GET /api/dispatch/batches/{batch_id}/orders
POST /api/dispatch/batches/{batch_id}/orders
PUT /api/dispatch/orders/{order_id}
DELETE /api/dispatch/orders/{order_id}
```

### 8.3 Driver APIs

```text
GET /api/drivers
POST /api/drivers
PUT /api/drivers/{driver_id}
```

### 8.4 Vehicle APIs

```text
GET /api/vehicles
POST /api/vehicles
PUT /api/vehicles/{vehicle_id}
```

### 8.5 Generate Plan API

```text
POST /api/dispatch/batches/{batch_id}/generate
```

Expected behavior:

1. Load orders, drivers, vehicles, and config for the batch
2. Convert records into existing dispatch optimizer models
3. Call the existing `DispatchEngine`
4. Save `plans`
5. Save `order_assignments`
6. Save `exceptions`
7. Return the dispatch result to the frontend

Response shape:

```json
{
  "plans": [],
  "order_assignments": [],
  "exceptions": []
}
```

### 8.6 Manual Assignment API

```text
PUT /api/dispatch/assignments/{assignment_id}
```

Supported update fields:

```json
{
  "driver_id": 1,
  "vehicle_id": 101,
  "manual_reason": "Customer requested this driver"
}
```

Expected behavior:

1. Reject update if batch is locked
2. Update assignment
3. Mark `assignment_source` as `MANUAL`
4. Save `manual_reason`
5. Write audit log
6. Run basic validation
7. Return updated assignment

### 8.7 Export API

```text
GET /api/dispatch/batches/{batch_id}/export
```

Recommended export formats:

1. Excel
2. CSV
3. Print-friendly HTML

## 9. Frontend Requirements

### 9.1 Dispatch Batch Selector

The frontend should allow office staff to:

1. Select a dispatch date
2. Create a new dispatch batch
3. Open an existing draft batch
4. Open a locked historical batch
5. See the current batch status

### 9.2 Manual Order Entry

The order entry page should support:

1. Add order
2. Edit order
3. Delete order
4. Save draft
5. Validate required fields

Required order fields:

```text
customer_name
delivery_address
suburb
postcode
urgency
pallet_count
bag_count
kg_count
window_start
window_end
```

Optional order fields:

```text
designated_driver_id
notes
```

### 9.3 Driver and Vehicle Review

The frontend should allow office staff to review and edit:

1. Driver availability
2. Driver shift time
3. Driver preferred zones
4. Vehicle availability
5. Vehicle capacity

### 9.4 Generate Plan

The frontend should call the backend API to generate the dispatch plan.

The button should be disabled when:

1. Batch is locked
2. Required order fields are missing
3. No available drivers
4. No available vehicles

### 9.5 Driver Assignment Summary

The primary result view should remain:

```text
Driver → Vehicle → Orders
```

Each driver section should show:

1. Driver name
2. Vehicle rego
3. Assigned order count
4. Total load summary
5. Urgent order indicator
6. Order list

Each order row should show:

1. Order ID
2. Customer name
3. Address
4. Suburb
5. Postcode
6. Zone
7. Urgency
8. Notes
9. Assignment source: AUTO or MANUAL

### 9.6 Exceptions and Warnings

The frontend should show:

1. Unassigned orders
2. Unknown postcode / zone mapping issues
3. No feasible vehicle
4. No feasible driver
5. Capacity warning
6. Locked batch warning
7. Manual override warning

### 9.7 Manual Adjustment UI

Each assignment row should allow:

1. Change driver
2. Change vehicle
3. Add manual reason
4. Save change

Manual changes should be visually marked.

Example display:

```text
Manual change
Reason: Customer requested this driver
```

### 9.8 Lock UI

The frontend should show a clear lock action:

```text
Lock Dispatch Plan
```

Before locking, show confirmation:

```text
After locking, this dispatch plan will become read-only. You can still view and export it, but you cannot edit orders or assignments.
```

After locking:

1. Show locked badge
2. Disable order editing
3. Disable assignment editing
4. Disable regenerate button

## 10. Validation Rules

### 10.1 Required Field Validation

Orders should require:

```text
customer_name
delivery_address
postcode
window_start
window_end
```

Drivers should require:

```text
driver_id
name
shift_start
shift_end
```

Vehicles should require:

```text
vehicle_id
rego
vehicle_type
```

### 10.2 Business Validation

The system should warn when:

1. Postcode cannot be mapped to a zone
2. Order is unassigned
3. Urgent order is unassigned
4. Vehicle capacity may be exceeded
5. Driver is unavailable
6. Vehicle is unavailable
7. Manual assignment conflicts with locked status
8. Manual assignment has no reason

Warnings should not always block saving, but locked dispatch plans should block all edits.

## 15. Development Guardrails

When implementing this MVP:

1. Do not change the existing top-level output contract.
2. Do not change scoring weights unless explicitly approved.
3. Do not change zone behavior unless explicitly approved.
4. Do not remove existing CLI behavior.
5. Do not remove existing frontend result views.
6. Keep backend dispatch engine as the authoritative planner.
7. Keep changes small and reviewable.
8. Add tests for every new business behavior.
9. Do not commit secrets, API keys, `.env` files, credentials, cache files, or local backups.
10. If locked plan behavior is changed, update tests and documentation together.

---

## 16. Summary

The Office Dispatch Workbench MVP should focus on making the existing dispatch optimizer usable by office staff.

The first version should solve one clear business problem:

```text
Office staff can manually enter daily delivery orders, generate driver and vehicle assignments, adjust the result, save it, lock it, and review it later.
```

This MVP should not try to become a full ERP system.

The priority is:

```text
Save data
Generate assignments from backend
Allow manual adjustment
Lock final result
Keep historical records
```
